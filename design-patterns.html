Address lookup

  • The example below uses 'unobtrusive JavaScript' to present address / postcode lookup functionality
  • The lookup functionality is considered 'value-added' and is therefore only presented when JavaScript is available
  • The standard address fields can be completed with JavaScript off
  • Note the following code and components:
    • The body element has a 'no-js' class <body class="no-js">
    • The address lookup div is hidden using css (i.e. the div is hidden when JavaScript is not available) .no-js div#address-lookup{ display: none; }
    • JQuery is used to remove the 'no-js' class (i.e. the div is shown when JavaScript is available) // remove the no-js class
      $('body').removeClass('no-js');
    • JQuery is used to show the select address input <script type="text/javascript">
      $(document).ready(function () {
        $('div#select-address').hide();
        $('button#find-address').click(function () {
          $('div#select-address').fadeIn(800);
        });
      });
      </script>
  • Please note that the font size of <legend> has been reduced to provide hierarchical presentation in this document. The font size as defined on the forms section of the components page should used.

Progress trackers

  • Used to guide the visitor through a number of steps to complete a specific process, typically a multi-step form or a checkout/payment process
  • Must be horizontally centered above the primary heading (<h1>)
  • Must not wrap to more than one line
  • Should be considered if there are 3 or more steps
  • If there are a large number steps (eg 8 or above) then you should reconsider the process/journey
  • If the page/window title and primary heading are repeated on each step of a multi-step form then the current step and total steps must be included <title>Step 2 | Order a birth certificate | nidirect</title>
    <h1 id="skip-dest">Order a birth certificate (Step 2 of 7)</h1>
  • The progress tracker can be hidden from view on the small and/or medium layouts if there is not enough screen real estate and the primary heading contains both the current and total step numbers (eg Step 2 of 6)
  • Must use the visual styling as per the examples below

Labelled tracker

A labelled step tracker is preferred when:

  • all steps can be easily defined using one or two words
  • visitors can easily guess the purpose of each step from its label
  • the steps follow a familiar or known convention (eg a checkout or payment process)
  • there are a small number of steps (eg 5 or less)
  • there is enough screen real estate to present each step without the progress tracker looking cluttered, confusing or wraps to additional lines
Examples and code for labelled progress trackers are not available on small screen devices.

Numbered tracker

A numbered step tracker is preferred when:

  • the individual steps cannot be easily defined using one or two words
  • visitors cannot easily guess the purpose of each step from its label
  • the steps do not follow a familiar or known convention
  • there are a large number of steps (eg 6 or more)

Date controls

Not all dates are the same. Choose a format appropriate for the kind of date you're asking about.

Memorable dates

For capturing memorable dates like dates of birth, the simplest option is to provide inputs to select the day and month, and then a text field to type in the year.

Approximate dates

If you don't need an exact date and users are likely to struggle to remember it (for example the date you started your first job), make sure you allow users to enter approximate dates like 'June 1996'.

Calendar controls

Calendar controls are only appropriate for near past or future dates, where the day of the week is relevant. Their main use case is for appointment booking.

Formatting dates

The default date format to use is this one:

  • 8 July 2014

Periods of time should be formatted like this:

  • 8 July to 9 August

Error Pages

  • every error, regardless of who is to blame, results in a bad user experience
  • effectively designed error pages can help to nullify that experience
  • error messages should sound like they are written by humans - for humans
  • where possible, an error page should give the user options on what to do next

Not found (404) error message

The 404 or 'not found' error message is an HTTP standard response code indicating that the client was able to communicate with a given server, but the server could not find what was requested. The web site hosting server will typically generate a "404 Not Found" web page when a user attempts to follow a broken or dead link

Internal server (500) error message

The 500 or 'internal server' error message is an error generated by the Web server. It basically means something has gone wrong, but the server cannot be more specific about the error condition in its response to the client. In addition to the 500 error notified back to the client, the Web server should generate some kind of internal error log which gives more details of what went wrong.

Session error message

The session error message will appear after a defined period of client inactivity, or in cases when the user attempts to click the back button after a transaction is complete. Guidance provided by OWASP on session expiration recommends that the period of client inactivity should be dependent on how critical the web application and its data are, and can range from 2 to 5 minutes for high-value applications to 15 to 30 minutes for low risk applications.

General error message

The general error message is designed to cover 'catch-all' errors not covered by either the 404 or 500 error messages.
